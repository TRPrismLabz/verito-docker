# Verito Docker — Deploy in 60 Seconds

One-click Docker deployment for [Verito](https://github.com/PrismLabz/verito), the open-source MCP security proxy for AI agents.

This repo handles deployment and configuration. For the source code, see [verito-core](https://github.com/PrismLabz/verito).

---

## Quick Start

```bash
git clone https://github.com/PrismLabz/verito-docker.git
cd verito-docker
chmod +x setup.sh && ./setup.sh
```

That's it. Open **http://localhost:8000**.

### What `setup.sh` does

1. Checks Docker and Docker Compose are installed and running
2. Copies `.env.example` → `.env` with an auto-generated secret key
3. Pulls the Verito image and starts the stack (Verito + Redis)

---

## Manual Setup

If you prefer to do it yourself:

```bash
git clone https://github.com/PrismLabz/verito-docker.git
cd verito-docker
cp .env.example .env
```

Edit `.env` with your keys (see below), then:

```bash
docker compose up -d
```

## Configuration

Edit `.env` after running setup:

| Variable | Required | What it does |
|---|---|---|
| `APP_SECRET_KEY` | Yes | Session security. Auto-generated by `setup.sh` |
| `NANGO_SECRET_KEY` | No | Enables real OAuth for Gmail, Slack, GitHub via [Nango](https://nango.dev). Leave blank for demo mode |
| `TELEGRAM_BOT_TOKEN` | No | Enables Telegram approval pings for write actions |
| `TELEGRAM_CHAT_ID` | No | Your Telegram chat ID (get from @userinfobot) |
| `REDIS_URL` | Auto | Points to the Redis container. Don't change unless using external Redis |
| `TOKEN_TTL_SECONDS` | No | How long token maps live. Default: 1800 (30 min) |

### Setting up Nango (Real OAuth Connections)

1. Sign up at [nango.dev](https://nango.dev)
2. Create integrations for Gmail, Slack, GitHub
3. Copy your secret key into `NANGO_SECRET_KEY` in `.env`
4. Restart: `docker compose restart verito`

Without Nango, Verito runs in **demo mode** with built-in sample data.

### Setting up Telegram Approvals

1. Message [@BotFather](https://t.me/BotFather) on Telegram → create a bot → copy the token
2. Message [@userinfobot](https://t.me/userinfobot) → copy your chat ID
3. Add both to `.env`
4. Restart: `docker compose restart verito`

Now when an agent tries a write action (send email, etc.), you get a Telegram ping.

## Connect Your Agent

After Verito is running, visit **http://localhost:8000/setup** for copy-paste configs.

### Claude Desktop

Add to `~/Library/Application Support/Claude/claude_desktop_config.json` (macOS) or `%APPDATA%\Claude\claude_desktop_config.json` (Windows):

```json
{
  "mcpServers": {
    "verito-refinery": {
      "command": "docker",
      "args": [
        "run", "-i", "--rm",
        "-v", "verito_vault:/app/data",
        "verito/refinery:latest",
        "python", "-m", "src.mcp"
      ]
    }
  }
}
```

Restart Claude Desktop. Verito tools appear in the tool picker.

### Cursor IDE

**Settings > Features > MCP > Add New Server** — paste the same JSON.

### OpenClaw

Add to `/config/agents/mcp_servers.json`:

```json
{
  "mcpServers": {
    "verito-refinery": {
      "command": "docker",
      "args": [
        "run", "-i", "--rm",
        "-v", "verito_vault:/app/data",
        "-e", "NANGO_SECRET_KEY=your_key_here",
        "-e", "REDIS_URL=redis://your-redis:6379",
        "verito/refinery:latest",
        "python", "-m", "src.mcp"
      ]
    }
  }
}
```

### n8n / LangChain / HTTP

Point directly at the running container:

```bash
curl -X POST http://localhost:8000/api/proxy/call \
  -H "Content-Type: application/json" \
  -d '{"tool_name": "secure_gmail_search", "arguments": {"query": "from:lawyer"}}'
```

## Commands

```bash
# Start
docker compose up -d

# Stop
docker compose down

# Logs
docker compose logs -f verito

# Restart after .env changes
docker compose restart verito

# Rebuild from source (if using local core repo)
docker compose up -d --build
```

## What's in the Stack

| Service | Image | Purpose |
|---|---|---|
| `verito` | `ghcr.io/prismlabz/verito:latest` | FastAPI app + MCP server + dashboard |
| `redis` | `redis:7-alpine` | Persistent token map storage across restarts |

## Architecture

```
┌──────────────┐     ┌──────────────────────┐     ┌─────────────┐
│  Your Agent   │     │   Docker Container    │     │ Gmail/Slack  │
│  (Claude,     │─MCP─│                       │─────│ GitHub APIs  │
│   Cursor,     │     │  Verito (port 8000)   │     │              │
│   OpenClaw)   │◄MCP─│  + Redis (port 6379)  │◄────│  Via Nango   │
│               │     │                       │     │  OAuth       │
│  Sees only:   │     │  PII scrubbed before  │     │              │
│  [PERSON_1]   │     │  reaching the agent   │     │              │
└──────────────┘     └──────────────────────┘     └─────────────┘
```

## Security Notes

- `.env` is in `.gitignore` — your keys never reach GitHub
- Redis data is in a Docker volume, not on disk
- The container runs with minimal permissions
- Write actions are blocked by default and require approval
